# Autodata vs Agent Swarm 对比分析

---

## 1. 定位差异

| 维度 | Meta Autodata | Agent Swarm |
|------|--------------|-------------|
| **本质** | 数据质量方法论 + 数据生产流水线 | Agent 编排平台 + 可视化管理工具 |
| **核心问题** | "如何自动生产高质量训练数据" | "如何管理和协调多个 Agent 执行任务" |
| **层次** | 具体的数据合成算法/流程 | 通用的 Agent 编排基础设施 |
| **类比** | 一台精密的数据加工机床 | 一个管理多条生产线的工厂中控室 |

## 2. 架构差异

| 维度 | Meta Autodata | Agent Swarm |
|------|--------------|-------------|
| Agent 角色定义 | 固定 5 角色体系（Challenger/Solvers/Judge/Orchestrator） | 用户自定义任意角色 |
| 数据流 | 严格的线性 + 反馈循环 | 灵活的流水线编排 |
| 质量控制 | **内建**（weak-strong solver 区分度机制） | **外挂**（质检 Agent，用户自行设计 prompt） |
| 自优化 | 有（元优化外循环） | 无（Agent prompt 完全依赖用户手动调优） |
| 执行引擎 | 直接 API 调用多个 LLM | Claude Agent SDK + Vercel AI SDK |
| 并发模型 | 每篇论文独立的串行处理 | 最多 10 个 Task 并发 |
| 状态管理 | 无持久化 | JSON 文件持久化 + WebSocket 实时推送 |

## 3. 数据质量控制差异（最核心的不同）

### Autodata 的质量控制：闭环系统
- 有一套**可量化的验收标准**（weak ≤ 65%, gap ≥ 20%）
- 通过**对抗性验证**（weak vs strong solver）确保数据区分度
- **自动迭代**：不达标就自动修改 prompt 重新生成
- **元优化**：连"如何造数据"这件事本身也被优化

### Agent Swarm 的质量控制：开环
- 质检依赖用户定义的 Agent prompt（"质检专家"）
- 没有内建的验收标准或自动化质量度量
- 迭代改进需要人工判断
- 没有"造数据流程"本身的优化机制

## 4. 互补性分析

| Agent Swarm 优势 | Autodata 缺失 |
|------------------|---------------|
| 可视化看板界面 | 无（命令行/日志） |
| 人工干预能力 | 无（全自动） |
| 多任务并行 | 串行处理 |
| 多模型支持（6+ Provider） | 固定模型组合 |
| WebSocket 实时推送 | 无 |

| Autodata 优势 | Agent Swarm 缺失 |
|---------------|-----------------|
| weak-strong solver 对抗验证 | 无质量验证机制 |
| 迭代式数据精炼 | 开环质检 |
| 结构化评分标准（Rubric） | 无 |
| 元优化（自动改进 prompt） | 无 |

## 5. 融合价值

将 Autodata 的数据质量方法论集成到 Agent Swarm，用户将获得：
- **Autodata 级别的数据质量保障**（weak-strong solver 验证）
- **Agent Swarm 级别的管理体验**（可视化、人工干预、多模型）
- 两者互补，不是替代关系
