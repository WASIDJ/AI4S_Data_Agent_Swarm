## 智能体 A：SDK 会话管理和数据流架构分析（sdk-analyst）

### A1. SDK 调用生命周期

**`startQuery()` 函数**（`server/sdk/queryWrapper.ts:312-358`）：

```typescript
const stream = sdkQuery({
  prompt: task.description,          // Task description 作为用户 prompt
  options: {
    systemPrompt: {
      type: "preset",
      preset: "claude_code",         // Claude Code 内置系统提示
      append: agent.prompt,          // Agent prompt 追加在末尾
    },
    cwd: projectDir,
    maxTurns: task.maxTurns,
    maxBudgetUsd: task.maxBudgetUsd,
    canUseTool,                       // 工具权限回调
    permissionMode: "default",
    // 通过 applyAgentOverrides 注入：
    model: agent.model,
    env: { ANTHROPIC_API_KEY, ANTHROPIC_AUTH_TOKEN, ANTHROPIC_BASE_URL },
  },
});
```

**会话生命周期**：
- 创建：`sdkSessionManager.startTask()` → `startQuery()` 或 `provider.startQuery()`
- 恢复：`resumeQuery(sessionId, message, ...)` → `options.resume = sessionId`
- 终止：`stopTask()` → `abortController.abort()` + 清理

### A2. 事件采集（双通道）

1. **主通道**：SDK AsyncGenerator → `consumeSDKStream()` → `processSDKMessage()` → Event
2. **补充通道**：Hook `POST /event` → `eventProcessor.processEvent()`

**消息转换**：`messageParser.ts` 负责 SDK Message → Event 的映射（SDKInit/SDKAssistant/SDKResult）。

**广播时机**：所有状态变更都触发 `broadcast()`（task:update, agent:update, event:new, tool:approval, notification）。

### A3. 扩展点分析

| 问题 | 结论 |
|------|------|
| 单 Task 内编排多个子 Agent？ | **不支持**。Task:Agent = N:1 绑定，无子任务机制 |
| systemPrompt.append 最大长度？ | 无硬限制，建议 ≤ 20,000 字符（留上下文空间） |
| 动态修改 Agent prompt？ | **不支持**。query() 调用时一次性传入 |
| Pipeline 编排能力？ | 只是 Task 模板工厂，**不是编排引擎**（无自动执行、条件分支、重试） |

## 智能体 B：数据模型和存储层分析（data-analyst）

### B1. Agent 完整字段

关键字段：`model`, `provider`, `apiKey`, `apiBaseUrl`, `prompt`, `maxTurns`, `maxBudgetUsd`

已支持 7 种 Provider：`claude | kimi | glm | minimax | codex | openai | deepseek`

### B2. Task 完整字段

关键字段：
- `pipelineType?: "qa" | "scievo"` — 已有流水线类型标记
- `parentTaskId?: string` — 已有链式关联
- `inputFiles?: string[]` — 已有输入文件列表
- `sessionId?: string` — Task:Session = 1:1 映射

状态机：`Todo → Running → Done / Stuck / Cancelled`
- Agent 单任务约束：`getActiveTaskForAgent()` 确保一个 Agent 同时只执行一个 Task
- 系统并发上限：`MAX_CONCURRENT_TASKS = 10`

### B3. 存储层能力

三层并发安全：
1. **proper-lockfile** — 文件级排他锁
2. **p-queue** — 写入队列（concurrency: 1）
3. **safeWrite** — 原子写入（.tmp → rename）

Schema 迁移机制：`_schema_version` 字段 + 递增迁移函数（已有 v1→v2 先例）。

### B4. 扩展评估

| 扩展需求 | 复杂度 | 说明 |
|---------|--------|------|
| 质量评分字段 | **低** | 纯字段扩展，updateTask 支持任意 patch |
| 迭代历史 | **中** | 需要迭代组聚合查询 |
| Rubric 评分标准 | **中** | 需要新 Store + Route |
| Prompt 版本管理 | **低-中** | 新 Store 或 Agent 内嵌 |
| 验证任务关联 | **中** | 需要关联表或 TaskRelation |

## 智能体 C：架构设计方案（architect）

### C1. 核心设计决策

1. **不走 pipeline.ts 扩展**，新增独立 `autodataService.ts` + `autodata.ts` 路由
   - 原因：autodata 是"动态创建下一步"，与现有 pipeline 的"一次性创建"本质不同
   - 最小侵入点：`sdkSessionManager.finalizeTask()` 末尾加一行判断

2. **全部 4 个角色用 AISDKProvider**
   - Challenger/Judge 也是纯 LLM 调用，不需要 Claude Code 工具能力
   - 用 ClaudeProvider 会浪费大量 token（system prompt + 工具定义开销）

3. **Phase 1 只做单轮验证，不做迭代**
   - Challenger 生成 → Weak/Strong Solver 验证 → Judge 评分 → 通过/拒绝

### C2. 五阶段方案

| 阶段 | 新增文件 | 修改文件 | 新增代码 | 复杂度 |
|------|---------|---------|---------|--------|
| Phase 1: Pipeline 模板 | 2 | 3 | ~290 行 | 低 |
| Phase 2: 数据模型扩展 | 1 | 4 | ~250 行 | 低 |
| Phase 3: 验证 + 迭代循环 | 3 | 3 | ~600 行 | 高 |
| Phase 4: 前端可视化 | 4 | 4 | ~500 行 | 中 |
| Phase 5: 元优化 | 3 | 4 | ~450 行 | 高 |
| **合计** | **13** | **18** | **~2090 行** | |

### C3. 依赖关系

```
Phase 1 (MVP) → Phase 2 (数据模型) → Phase 3 (迭代循环)
                                        ↓
                                   Phase 4 (前端)
                                        ↓
                                   Phase 5 (元优化)
```

## 智能体 D：可行性评估（feasibility-reviewer）

### D1. 关键可行性问题

| 问题 | 结论 |
|------|------|
| 多 Agent 协同？ | 用多 Task 编排（非单 Task 内多 Agent），现有机制可行 |
| 迭代循环？ | 新增 autodataService 作为编排引擎，finalizeTask 回调触发 |
| Solver 调用？ | AISDKProvider 的 streamText() 直接可用 |
| 存储支撑？ | 轨迹数据用 JSONL 追加写入，不影响现有 JSON 存储 |

### D2. 风险与降级

| 风险 | 等级 | 降级方案 |
|------|------|---------|
| Judge 输出格式不稳定 | 中 | structured output 约束 + fallback 正则 |
| 迭代循环中断 | 中 | 超时检测 + 手动重试端点 |
| Token 消耗 | 低 | Phase 1 先做单轮，验证价值后再开多轮 |
| 编排 Agent 上下文溢出 | 中 | 限制 maxIterations=3，每轮只保留摘要 |

### D3. 最小可行方案（2 天）

只做 Phase 1 + Phase 2 合并：
- 新增 autodataService（~200 行）+ autodata 路由（~60 行）
- Task 新增 autodataMeta 和 qualityScore 字段（~30 行）
- sdkSessionManager 加回调（~5 行）
- **总计 ~327 行代码改动**
